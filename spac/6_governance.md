# 10. 데이터 거버넌스 (Data Governance)

분류체계 및 데이터의 신뢰성, 보안, 추적성을 보장하기 위해 다음과 같은 거버넌스 기능을 포함한다.

### 10.1 권한 및 역할 관리 (RBAC)
데이터와 스키마에 대한 접근을 제어하기 위해 4가지 주요 역할을 정의한다.

| 역할 | 권한 범위 | 설명 |
|---|---|---|
| **System Admin** (시스템 최고 관리자) | 전역 (Global) | 새로운 도메인 생성, 글로벌 시스템 설정, 전체 권한 관리 |
| **Domain Steward** (도메인 관리자) | 특정 도메인 단위 | 담당 도메인 내의 분류 노드(Classification Node) 관리, 필드 스키마 정의 및 수정 |
| **Data Editor** (데이터 편집자) | 특정 노드 단위 | 할당된 노드 및 하위 노드의 레코드(Record) 생성, 수정, 상태 변경 |
| **Viewer** (조회자) | 노드/레코드 단위 | 데이터 조회 전용. 민감 데이터(암호화 필드 등)에 대한 마스킹 해제 권한은 별도 부여 필요 |

> **설계 포인트:** 권한 엔티티(`role`, `user_role_mapping`)를 도입하여 각 사용자에게 '도메인 범위' 또는 '노드 범위'로 권한을 할당한다.

### 10.2 변경 이력 추적 (Audit Trail)
누가, 언제, 어떤 데이터를 변경했는지 완벽하게 추적할 수 있도록 데이터 및 스키마 수준의 감사 로그를 기록한다.

1. **스키마 감사 로그 (Schema Audit Log)**
   - 필드 추가/수정/논리삭제, 노드 생성/이동/삭제 등의 이벤트 기록
   - 기록 내용: 변경자 ID, 변경 일시, 대상 엔티티, 이전 상태(Before), 변경 후 상태(After)
2. **데이터 감사 로그 (Record Audit Log) - 구현 완료**
   - **백엔드 (`RecordHistory`)**: 데이터 생성(`CREATE`), 수정(`UPDATE`), 삭제(`DELETE`) 시 변경 전후 데이터를 JSONB 스냅샷으로 저장한다. 이력에는 변경자 UUID, 반영 일시(`changedAt`), 결재 문서 식별자(`approvalRequestId`)가 포함되어 승인 내역과 데이터 변경을 연결한다.
   - **프론트엔드 (이력 테이블)**: 과거 변경 내역을 내부 시스템 키 대신 번역된 필드명을 사용하여 가독성 높은 표 형태로 렌더링한다.
   - **스냅샷 보기**: 과거 특정 이력을 클릭하면, 해당 시점의 원본 데이터를 읽기 전용 팝업으로 복원하여 확인할 수 있다.
   - **결재 내역 역추적**: 이력 행 내의 [결재 내역] 버튼을 통해, 데이터를 변경하게 된 워크플로우 기안자, 결재 진행 과정, 코멘트 등을 즉시 확인할 수 있다.

### 10.3 스키마 및 데이터 변경 승인 워크플로우 (Approval Process)
운영 환경에서 스키마(노드/필드)뿐만 아니라 **실제 데이터(레코드)**가 무분별하게 변경되는 것을 방지하기 위해 통일된 승인 워크플로우를 거친다.

**[데이터(Record) 워크플로우] - 구현 완료**
1. **생성/수정/삭제 요청 (Pending):** Data Editor가 레코드를 조작하면 즉시 데이터가 반영되지 않고, 이전/이후 변경 사항(Changes)을 담은 `ApprovalRequest`(상태: `PENDING`) 데이터가 생성된다.
2. **결재 단계 진행 (Steps):** 라우팅 룰에 따라 순차적인 `ApprovalStep`이 생성되며, 할당된 결재권자들은 전용 "Approvals" 페이지에서 결재를 진행한다. 결재자는 화면상에서 Before/After 차이를 확인하고 **의견(Comment)** 을 기재한 후 승인 또는 반려를 선택한다.
3. **최종 승인(Approve) 및 반영:** 모든 결재 단계가 승인 처리되면, `RecordHistory`에 이력을 남김과 동시에 실제 원본 `Record` 테이블에 데이터가 최종 반영(`ACTIVE` 처리 또는 `DELETE`)된다.
4. **반려(Reject):** 결재 단계 중 하나라도 반려되면, 해당 상신 건 전체가 `REJECTED` 처리되며 원본 데이터는 수정 전 상태를 유지한다.

**[스키마(Schema) 워크플로우]**
1. **Draft 상태:** 도메인 관리자가 필드 추가/노드 이동을 수행하면 즉시 반영되지 않고 변경 요청(`approval_request`)이 생성됨.
2. **Review & Approve:** 데이터 거버넌스 위원회 또는 시스템 관리자가 검토 후 승인(Approve) 혹은 반려(Reject)함.
3. **Publish:** 승인 시 실제 스키마 테이블(`field_definition`, `classification_node`)에 변경 사항이 반영되고 EffectiveField가 재계산됨.

### 10.4 데이터 마스킹 및 개인정보 보호
- `is_encrypted=true` 인 필드(예: 주민등록번호, 계좌번호 등)는 DB에 암호화되어 저장될 뿐만 아니라, 애플리케이션 레벨에서도 제어됨.
- 조회 API 호출 시, 사용자의 권한(권한 설정 내 `can_unmask_pii` 속성)에 따라 마스킹 처리된 값(`******`)을 반환하거나 복호화된 평문을 반환함.
